Known bugs:
- Cross-block liveness for inlined constants: when many inlined constant values
  from different early blocks need to survive into a block with many calls,
  the per-block regalloc can assign them the same register because it doesn't
  see cross-block interference.
  Root cause: inlining constfolds inc(10)->11 and dbl(20)->40 into separate
  early blocks (block13, block15). Both constants flow via block params into
  block16 which has 6 calls. The per-block allocator for each early block
  assigns its constant to RAX (no interference within that block). But both
  RAX values must survive into block16, and the cross-block spill/reload
  mechanism doesn't detect that they collide. In the generated asm:
    mov eax, 0xb    ; 11 (inc(10))
    mov eax, 0x28   ; 40 (dbl(20)) -- overwrites 11!
  Further investigation revealed this is actually a single-block bug too:
  8 noinline calls with comparisons after also fails (no inlining needed).
  The deeper root cause: the allocator's spill/remat pass shortens Iconst
  live ranges by rematerializing them near their later uses (X86Sub). After
  remat, the original Iconst's live range only extends to its EffectfulUse
  marker (call arg), NOT across subsequent call barriers. The call clobber
  phantom interferences only apply to VRegs live at the call point, so after
  remat the original Iconst no longer interferes with caller-saved phantoms
  and gets RAX -- which is then clobbered by later calls before the actual
  call that uses it executes.
  Attempted fix (reverted): injecting committed-register precoloring from
  predecessor blocks caused allocator hangs (EffectfulUse dummies in
  block_live_out prevented spill convergence).
  The fix likely requires: (a) making the spill pass aware that call-arg
  VRegs cannot be rematerialized away from their EffectfulUse position, OR
  (b) recomputing call points after each spill round so clobber phantoms
  track the modified schedule, OR (c) extending EffectfulUse to also keep
  call args in block_live_out so they survive past all call barriers.
  Simpler reproducer (no inlining, pure single-block):
    __attribute__((noinline)) int id(int x) { return x; }
    int main() {
        int a=id(1); int b=id(2); int c=id(3); int d=id(4);
        int e=id(5); int f=id(6); int g=id(7); int h=id(8);
        if (a!=1) return 1; if (b!=2) return 2; // ... fails at a!=1
    }
